EaseMyCollab

A Django REST Framework platform connecting Brands and Influencers for campaign collaboration, application management, and mutual review.

Django 5.x + DRF JWT Auth SQLite / PostgreSQL 5 Core Modules

System Architecture

High-Level

EaseMyCollab is structured as a Django monolith with 5 distinct apps. The frontend is served via Django templates and the API layer is built with Django REST Framework. JWT tokens handle stateless authentication.

Auth Strategy
JWT via djangorestframework-simplejwt. Access + Refresh token pair issued on login. Refresh tokens enable session extension without re-login.
Signal-Driven Profiles
On User creation, a post_save signal auto-creates either a BrandProfile or InfluencerProfile based on role.
Role-Based Access
All protected endpoints check request.user.role. Only brands create campaigns; only influencers apply. Reviews require accepted collaboration status.

Accounts Module

Module

Handles all user lifecycle operations: registration, login, password reset, and profile read/write. Acts as the central identity service for both Brand and Influencer users.

What this module does

  • Custom User Model extends AbstractUser, adding role (brand/influencer) and phone fields. Email is enforced as unique.
  • Signal-driven profile creation — on User save, Django signals fire automatically to create either a BrandProfile or InfluencerProfile depending on the role.
  • JWT-based login — credentials are validated with Django's authenticate(), and a simplejwt access + refresh token pair is returned.
  • Forgot Password — generates a 6-digit OTP, caches it for 5 minutes under key reset_otp_{email}, and emails it via SMTP.
  • Reset Password — validates OTP from cache, sets new password using set_password(), clears OTP from cache.
  • Unified Profile API — serves both Brand and Influencer profile data from a single endpoint based on request.user.role. Also computes average rating from the Review table.
  • Profile update — POST to the same profile endpoint updates the respective profile fields (name, bio, industry, social URLs).

How data flows (Registration)

Client POSTs credentials
→
RegisterView validates input
→
Checks username/email uniqueness
→
create_user() saves hashed password
→
post_save signal fires → creates BrandProfile or InfluencerProfile

Key logic in ProfileView

  • Optional ?user_id= query param — if provided, returns another user's public profile. Otherwise returns the authenticated user's own profile.
  • is_owner flag — set to True only when viewing own profile, controls whether the Edit button appears in frontend templates.
  • Ratings aggregation — queries Review.objects.filter(reviewee=target_user), aggregates average rating with Avg('rating'), fetches latest 5 reviews.
  • Graceful degradation — all review logic wrapped in try/except so profile still loads even if Review table doesn't exist yet.

Campaigns Module

Module

The core business module. Manages the full campaign lifecycle from creation to application management, status updates, and post-collaboration reviews.

What this module does

  • Campaign creation — only users with role=brand can create campaigns. Brand profile is automatically linked from request.user.brandprofile.
  • Campaign listing — public endpoint (no auth required) returning all campaigns with full field serialization.
  • My campaigns — brand-specific endpoint filtering campaigns by brand=request.user.brandprofile.
  • Delete campaign — verifies ownership before deleting to prevent unauthorized deletion.
  • Apply to campaign — influencer-only. Checks role, fetches/creates InfluencerProfile using get_or_create for safety, prevents duplicate applications with Application.objects.filter().exists() check.
  • View applicants — brands can see all applications for a given campaign ID. ApplicationSerializer enriches output with influencer name and user ID.
  • Update application status — accepts 'accepted' or 'rejected' values, updates the Application.status field and saves.
  • Post review — only allowed if Application.status is 'accepted'. Prevents duplicate reviews (same reviewer + application combo). Reviewer determines reviewee: if brand is reviewing, reviewee is the influencer's user, and vice versa.

Application Status State Machine

pending  Default on apply
→
accepted  Brand approves
or
rejected  Brand rejects
→
review enabled  After accepted

How data flows (Application)

  • Role guard — returns 400 immediately if request.user.role != 'influencer'.
  • Profile safety — uses get_or_create on InfluencerProfile to avoid 404 if profile doesn't exist yet.
  • Duplicate guard — filters by campaign + influencer combo before creating new application.
  • Creation — always sets status='pending' explicitly regardless of any request payload.

Review Logic (critical)

  • Mutual reviews — both brand and influencer can review each other for the same collaboration. Each review is tied to one application.
  • Reviewee determination — if request.user == application.campaign.brand.user → reviewee is influencer's user. Otherwise → reviewee is brand's user.
  • Duplicate prevention — checks Review.objects.filter(application=application, reviewer=request.user).exists() before creating.
  • Status gate — reviews only allowed when application.status == 'accepted'.

Collaborations Module

Module

A thin but important module that formalizes an accepted application into an active collaboration record. Currently provides basic accept + duplicate-guard logic.

What this module does

  • Accept application — creates a Collaboration record pointing to an Application. Sets status='active' and fills default terms.
  • Duplicate guard — uses Collaboration.objects.filter(application_id=application_id).exists() to prevent creating the same collaboration twice.
  • 1:1 relationship with Application — enforced at model level with OneToOneField, guaranteeing only one collaboration per application.

Design notes

  • Thin module by design — the Collaboration model exists as a separate concept to allow future expansion (add contract terms, milestones, payment tracking) without polluting the Application model.
  • Auth currently commented out — permission_classes is disabled on AcceptApplicationView. Should be re-enabled in production to restrict to brand users only.
  • Standard terms hardcoded — terms="Standard terms" is a placeholder; real implementation would accept terms from request data.

Files

serializers.pyCollaborationSerializer — minimal field exposure

Brands Module

Module

Manages brand profile data. Provides create/update for brand profile fields including industry, bio, website, and social media links.

What this module does

  • BrandProfile CRUD — POST to create or update a brand's profile fields. Uses get_or_create to safely handle first-time profile setup.
  • Auto-populated from signals — BrandProfile is auto-created (with username as name) when a brand User registers, via accounts' post_save signal. This view is for subsequent updates.
  • Social link storage — supports instagram_url, twitter_url, facebook_url, and youtube_url as URLFields.
  • Logo upload — ImageField with upload path logos/ under MEDIA_ROOT.

Key design point

  • Profile served through ProfileView in accounts — the BrandProfileView in this module is supplementary. The primary profile read/write endpoint (/api/profile/) lives in the accounts app and handles both brand and influencer profiles uniformly.
🎤

Influencers Module

Module

Manages influencer profile data including niche, bio, and social links. Mirrors the brands module for the influencer side.

What this module does

  • InfluencerProfile CRUD — GET to read (own or any user's profile via ?user_id), POST to update fields like full_name, niche, bio, social URLs.
  • Multi-user profile lookup — InfluencerProfileView supports fetching another user's profile by user_id. If not found as influencer, falls back to checking BrandProfile.
  • is_owner flag — returned as False when viewing someone else's profile, so the frontend can hide the Edit button.
  • Auto-created on registration — just like brands, InfluencerProfile is auto-created via post_save signal in accounts.

Key design point

  • Primary profile endpoint is in accounts — /api/profile/ from AccountsModule handles influencer profiles too. The views in this module are secondary/utility views.
  • Profile pic uploads — ImageField with path profiles/ under MEDIA_ROOT.

Auth APIs

REST Endpoints
POST /api/register/ Register a new user

Request Body

{ "username": "mynick", "email": "me@brand.com", "password": "pass123", "role": "brand" }

What happens

  • Checks username and email uniqueness separately, returns specific 400 errors for each
  • Creates User with hashed password via create_user()
  • post_save signal fires → auto-creates BrandProfile or InfluencerProfile
  • Returns 201 on success

Response (success)

{ "message": "Registration successful" }
POST /api/login/ Login and receive JWT tokens

Request Body

{ "username": "mynick", "password": "pass123" }

What happens

  • Validates via LoginSerializer, runs Django's authenticate()
  • On success, generates simplejwt RefreshToken and returns access + refresh tokens
  • Returns 401 on invalid credentials

Response

{ "message": "Login successful", "access": "eyJ...", "refresh": "eyJ...", "user": { "id": 1, "username": "mynick", "role": "brand" } }
POST /api/forgot-password/ Send OTP for password reset

What happens

  • Checks if email exists in the database, returns 404 if not
  • Generates random 6-digit OTP, stores in Django cache under reset_otp_{email} for 5 minutes
  • Sends email via SMTP configured in settings
{ "email": "me@brand.com" }
POST /api/reset-password/ Reset password using OTP

What happens

  • Retrieves stored OTP from cache, compares with submitted OTP
  • Returns 400 if OTP is invalid or expired (TTL exceeded)
  • Calls user.set_password(new_password) and saves
  • Deletes OTP from cache to prevent reuse
{ "email": "me@brand.com", "otp": "123456", "new_password": "newPass!" }

Campaign APIs

REST Endpoints
GET /api/campaigns/ List all campaigns — public
  • No authentication required — fully public endpoint
  • Returns all campaigns with all fields via CampaignSerializer
  • No pagination currently implemented
POST /api/create-campaign/ Create a campaign 🔒 brand only

Request Body

{ "title": "Summer Launch", "description": "...", "deadline": "2025-08-01", "budget_range": "10k-20k", "status": "open" }
  • Returns 403-equivalent message if user role is not 'brand'
  • brand field is set server-side from BrandProfile.objects.get(user=request.user) — not from request body
  • Returns validation errors from CampaignSerializer if any field is invalid
GET /api/my-campaigns/ Brand's own campaigns 🔒 brand only
  • Filters campaigns by brand=request.user.brandprofile
  • Returns 400 if non-brand user hits this endpoint
DEL /api/delete-campaign/<id>/ Delete a campaign 🔒 owner only
  • Verifies campaign.brand == request.user.brandprofile before deleting
  • Returns error response if ownership check fails
POST /api/apply/<campaign_id>/ Apply to a campaign 🔒 influencer only
  • Returns 400 if user role is not 'influencer'
  • Uses get_or_create on InfluencerProfile for safety
  • Returns 400 if already applied (duplicate check)
  • Returns 404 if campaign doesn't exist (via get_object_or_404)
  • Creates Application with status='pending'
GET /api/applicants/<campaign_id>/ View applicants for a campaign 🔒 auth required
  • Filters applications by campaign_id
  • Returns influencer_name and influencer_user_id via ApplicationSerializer for frontend profile linking
GET /api/influencer/my-applications/ Influencer's own applications 🔒 influencer only
  • Uses select_related('campaign__brand') to avoid N+1 queries
  • Returns campaign title, brand name, brand_user_id (for profile link), status, budget, and applied_at date
POST /api/application/<application_id>/update/ Update application status 🔒 auth required 
{ "status": "accepted" }  // or "rejected"
  • Only allows 'accepted' or 'rejected' as valid status values
  • Returns 400 for any other status string
  • Returns 404 if application_id does not exist
POST /api/post-review/ Submit a review 🔒 accepted collab only 
{ "application_id": 5, "rating": 4, "comment": "Great collaboration!" }
  • Returns 400 if application status is not 'accepted'
  • Returns 400 if reviewer has already submitted a review for this application
  • Determines reviewee automatically based on who is reviewing
  • rating must be 1–5 (enforced by MinValueValidator/MaxValueValidator)

Collaboration APIs

REST Endpoints
POST /api/accept-application/<application_id>/ Formalize an application into a collaboration
  • Checks if a Collaboration already exists for this application_id — returns error if so
  • Creates Collaboration with status='active' and terms='Standard terms'
  • OneToOne constraint at DB level also prevents duplicates at the data layer
  • Note: auth is currently disabled on this view — should be restricted to brand users in production

Profile APIs

REST Endpoints
GET /api/profile/?user_id=<optional> Get own or another user's profile 🔒 auth required
  • Without user_id: returns authenticated user's own profile
  • With user_id: returns that user's public profile data
  • Includes avg_rating, total_reviews, and latest 5 reviews regardless of role
  • is_owner=True only for own profile — controls frontend edit visibility

Response (Brand)

{ "id": 1, "role": "brand", "is_owner": true, "name": "Acme Co", "industry": "Fashion", "website": "acme.com", "bio": "...", "instagram": "...", "youtube": "...", "avg_rating": 4.5, "total_reviews": 12, "reviews": [...] }
POST /api/profile/ Update own profile 🔒 auth required
  • Automatically detects whether user is brand or influencer via request.user.role
  • Updates only the fields present in request.data (falls back to existing value)
  • Uses get_or_create to safely handle first-time update
{ "name": "New Name", "bio": "Updated bio", "instagram": "https://instagram.com/..." }
