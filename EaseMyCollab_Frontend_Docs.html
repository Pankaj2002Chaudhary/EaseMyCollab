Django templates + vanilla JavaScript + Fetch API. All pages are server-rendered HTML shells — the data is fetched client-side from the live Render deployment.

Frontend Architecture

How It Works

EaseMyCollab uses a server-rendered shell + client-side data pattern. Django renders the HTML page (with navbar, layout, and empty containers), and JavaScript fetch() calls fill in the live data from the API. There is no React, no Vue — just plain HTML templates with embedded JS.

Key design patterns used

  • No page reloads for data — all API data is fetched via fetch() after DOMContentLoaded fires. The HTML from Django is just a static shell.
  • Role-based UI branching — navbar and redirects differ for brands vs influencers, controlled by localStorage.getItem("role").
  • Optimistic UI — buttons are disabled and text changed immediately (e.g., "Applying...") before the API responds, then updated on success/failure.
  • Toast notifications — a fixed-position #toast div appears and auto-dismisses after 3 seconds instead of alert() (on home and profile pages).
  • Hard-coded Render URL — all fetch() calls point to https://easemycollab.onrender.com/api/... directly, even when served from the same domain.

Home Page

/ (root)

The public landing page. Shows a hero section, stat strip, and all live campaigns fetched from the API. Influencers can apply directly from this page.

What happens on page load

1
Django serves the HTML shell
Base template renders with empty #campaign-list container showing "Scanning for new opportunities..."
2
JS calls loadCampaigns() immediately
No DOMContentLoaded wrapper — runs as soon as the script tag executes at bottom of page.
3
GET /api/campaigns/ — no auth required
Public endpoint returns all campaigns. Response is JSON array of campaign objects.
4
Campaigns rendered into DOM
data.forEach(c => ...) creates a card div per campaign and appends it to #campaign-list. Also updates #totalCampaigns stat counter.
5
Express Interest button
Each card has a button calling applyToCampaign(id). If no token in localStorage, shows toast + redirects to /login/. Otherwise POSTs to /api/apply/:id/ with Bearer token.
GET https://easemycollab.onrender.com/api/campaigns/ No auth
Returns all campaigns. Rendered into campaign cards.
// Each item rendered as:
{ id, title, description, budget_range, status, deadline, ... }
POST https://easemycollab.onrender.com/api/apply/:id/ Bearer token required
No body needed. Campaign ID in URL. Returns 201 on success, 400 if already applied.

Key UI details

  • Stats strip — "Active Gigs" count is dynamically set to data.length. Other stats (1.2k+ creators, 500+ brands, ₹5M+ paid) are hardcoded placeholder values.
  • Button state management — on click, button is immediately disabled and text changes to "Applying...". On success: "✓ Applied". On failure: re-enabled with original text.
  • Error handling — network failures caught in try/catch, show "Network Error" toast. API errors show "Failed to apply" toast.
  • Category field — uses c.category || 'Collaboration' as fallback since Campaign model has no category field yet.

/login/   /register/
/login/ auth/login.html
  • Enter key support — keydown event listener calls handleLogin() on Enter keypress
  • Validation — checks both fields are non-empty before firing fetch
  • API call — POST /api/login/ with username + password as JSON

  • Role selector — segmented control (radio buttons styled as pills): "Influencer" selected by default, "Brand" as second option

  • API call — POST /api/register/ with all 4 fields including selected role
  • On success — shows green success message "Registration successful! Redirecting..." then navigates to /login/ after 1500ms
  • On failure — shows data.error in red error box (e.g. "Username already taken")

Brand Dashboard

/dashboard/

Brands see all their campaigns in a card grid. Clicking a campaign card expands it inline to show all applicants, with Accept/Reject buttons for pending ones.

How data loads

1
DOMContentLoaded → loadCampaigns()
GET /api/my-campaigns/ with Bearer token. Campaigns sorted by ID descending (newest first).
2
Each campaign card rendered with click handler
card.onclick = () => toggleApplicants(c.id). Cards contain an empty #applicants-{id} div.
3
Click → toggleApplicants(campaignId)
Accordion behavior: only one campaign's applicants shown at a time. Second click on same campaign collapses it.
4
GET /api/applicants/:id/ → renders each applicant
Shows influencer name. Pending applications get Accept/Reject buttons. Accepted/rejected ones show a status tag.
5
Accept/Reject → POST /api/application/:id/update/
On success, calls location.reload() to refresh the full page so status tags update. event.stopPropagation() prevents the click from bubbling to the campaign card toggle.
GET https://easemycollab.onrender.com/api/my-campaigns/ Brand's campaigns
Returns campaigns belonging to this brand. Bearer token identifies which brand.
GET https://easemycollab.onrender.com/api/applicants/:campaignId/ Lazy loaded on click
Fetched only when brand clicks a campaign card — not upfront. Returns applicant name + status.
POST https://easemycollab.onrender.com/api/application/:id/update/ Accept or reject 
{ "status": "accepted" }  // or "rejected"

Profile Page

/profile/   /profile/?user_id=X

Unified profile page for both brands and influencers. Supports two modes: viewing your own profile (with edit capability) or viewing someone else's public profile (read-only).

How it determines what to show

1
Parse URL for ?user_id param
const targetUserId = new URLSearchParams(window.location.search).get('user_id')
2
Build API URL with optional user_id
If targetUserId exists: /api/profile/?user_id=X. Otherwise: /api/profile/
3
API returns is_owner flag
If data.is_owner === true: show "Edit Details" button. If false: show nothing in the button row (read-only view).
4
Fields filled via fillFields(data)
Sets value on all inputs (name, industry, website, bio, instagram, youtube). Avatar letter set to first char of name.
5
Edit mode via enableEdit()
Removes disabled from all .custom-input elements. Swaps button to "Cancel" + "Save Changes".
6
Save via POST /api/profile/ with all field values
On success: shows toast "Profile updated!" and calls cancelEdit() to go back to read-only state and reload data.

Create Campaign Page

/create-campaign/

A centered form card for brands to publish new campaigns. Accessible only to logged-in brand users (frontend checks token; backend checks role).

How it works

  • Token check on submit — if localStorage.getItem("token") is null, shows alert and redirects to /login/.
  • Validation — title and description are required. Budget and deadline are optional (but sent if provided).
  • POST /api/create-campaign/ — sends title, description, budget_range, deadline, status="open" as JSON with Bearer token.
  • On success — button turns green "✓ Published", then redirects to /dashboard/ after 1000ms.
  • On failure — alert with stringified error data. Button re-enabled.
POST https://easemycollab.onrender.com/api/create-campaign/ Brand only 
{ title, description, budget_range, deadline, status: "open" }

Token Management

localStorage

JWT is stored client-side in localStorage. No refresh token logic is implemented on the frontend — when the access token expires (60 min), the user must log in again.

"token"
JWT access token from login response
eyJhbGciOiJIUzI1NiIs...
"role"
User role for UI branching
"brand" | "influencer"
"name"
Display name for navbar avatar
"Pankaj" — first char shown

How token is used

// Every protected fetch call looks like this:
const token = localStorage.getItem("token");

const res = await fetch("https://easemycollab.onrender.com/api/my-campaigns/", {
  headers: { "Authorization": "Bearer " + token }
});

// For POST with body:
const res = await fetch("https://easemycollab.onrender.com/api/create-campaign/", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": "Bearer " + token
  },
  body: JSON.stringify({ title, description, budget_range, deadline, status: "open" })
});

Logout

  • Navbar logout button — calls handleLogout() which does localStorage.clear() then redirects to /login/.
  • No server-side session invalidation — JWT is stateless; clearing localStorage is sufficient for the client. Token remains valid server-side until expiry (60 min).

Navbar Logic

navbar.html

The navbar is a standalone include file (navbar.html) included in every page via base.html. It dynamically builds its content based on localStorage using the updateNavbar() function.

flowchart TD LOAD["DOMContentLoaded\nupdateNavbar() fires"] TOKEN{localStorage\nhas token?} LOAD --> TOKEN TOKEN -->|Yes| LOGGED["Show:\nDashboard link\nCreate link (brand only)\nProfile avatar pill\nLogout button\nHamburger (mobile)"] TOKEN -->|No| GUEST["Show:\nLogin link\nJoin Now button\nNo hamburger"] LOGGED --> ROLE{role = 'brand'?} ROLE -->|Yes| BRAND_NAV["Dashboard → /dashboard/\nCreate → /create-campaign/"] ROLE -->|No| INF_NAV["Dashboard → /influencer/dashboard/\nNo create link"] style LOGGED fill:#1A2A1E,stroke:#34D399,color:#E8EAF0 style GUEST fill:#2A1A1A,stroke:#FB7185,color:#E8EAF0

Key details

  • Sticky navbar — position: sticky; top: 0; z-index: 1000 with backdrop-filter: blur(12px) for glass effect.
  • Avatar pill — shows first letter of name from localStorage.getItem("name"). Falls back to "U" if not set.
  • Active link highlight — compares window.location.pathname to each link's href and adds .active class on match.
  • Mobile hamburger — hidden for guest users (no token). Only shown when logged in. Toggles .nav-mobile visibility and animates the middle bar opacity.
  • Logout — calls handleLogout(): localStorage.clear() then window.location.href = "/login/".

How deployment works

1
Code pushed to GitHub
Render auto-detects the Python project from requirements.txt
2
Render runs build
pip install -r requirements.txt then python manage.py collectstatic --noinput
3
Render sets env vars
DATABASE_URL pointing to managed Postgres, DEBUG=False for production
4
Gunicorn starts via Procfile
Serves Django app over WSGI. WhiteNoise serves /static/ files directly from the process — no Nginx needed.
5
Settings automatically switch DB
if DATABASE_URL: → uses dj_database_url.config() for PostgreSQL. Else → falls back to SQLite for local dev.

Static files pipeline

  • Development — Django serves /static/ from static/ dir directly. CSS lives at static/css/style.css, JS at static/js/main.js.
  • Production collectstatic — python manage.py collectstatic copies all static files into staticfiles/ directory.
  • WhiteNoise — WhiteNoiseMiddleware in MIDDLEWARE (after SecurityMiddleware) serves files from staticfiles/ with compression and long-term caching via CompressedManifestStaticFilesStorage.
  • Template static tags — {% load static %} in templates generates correct URLs like /static/css/style.css which WhiteNoise intercepts.

⚠️ Security notes
  • SECRET_KEY is hardcoded — should be moved to an environment variable before production use.
  • Gmail App Password visible in code — should be in an env var (os.getenv('EMAIL_HOST_PASSWORD')).
  • OTP in LocMemCache — resets if the Render service restarts. Consider Redis for persistent OTP storage in production.
  • No HTTPS enforcement — SECURE_SSL_REDIRECT is not set. Render handles HTTPS at the edge.
